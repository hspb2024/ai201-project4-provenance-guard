# Provenance Guard

A content-attribution service. A creator submits text; the system estimates whether
it was written by a human or generated by AI using **two independent detection
signals**, combines them into one calibrated confidence score, attaches a
plain-English transparency label, records the decision in a structured SQLite audit
log, and lets creators **appeal** a result they believe is wrong.

The full design rationale — signals, uncertainty model, label variants, appeals
workflow, edge cases, and architecture diagrams — lives in [planning.md](planning.md).
This README is the canonical record of *what was built and why*.

---

## Architecture

```
POST /submit {text, creator_id}
        │  raw text
        ▼
  [Request layer: Flask + Flask-Limiter]  ── validate + rate-limit ──►
        │  raw text
        ├──────────────► [Signal 1: Groq LLM]   ── llm_score (0–1) + rationale ─┐
        │  raw text                                                             │
        └──────────────► [Signal 2: Stylometry] ── stylo_score (0–1) + feats ───┤
                                                                                ▼
                                                     [Scoring]  confidence = 0.6·llm + 0.4·stylo
                                                                → certainty, attribution bucket
                                                                                │
                                                                                ▼
                                                     [Label]  ⚑ / ✓ / ?  transparency text
                                                                                │
                        ┌────────────────────────────────────────────────────────┤
                        ▼                                                          ▼
                 [Audit log: SQLite]  <── structured row ──          Response {content_id,
                 content_id, both scores, confidence,                attribution, confidence,
                 attribution, label, status                          certainty, label, signals}

POST /appeal {content_id, creator_reasoning}
        ▼
  [Appeal handler]  status: classified → under_review, append audit row ──► GET /appeals (reviewer queue)
```

**Flow narrative.** On submission, the raw text fans out to both signals; their
scores combine into one `confidence` (the combined AI-likelihood), which is bucketed
into an attribution (`likely_ai` / `uncertain` / `likely_human`) and turned into a
transparency label. Everything is logged and returned with a `content_id`. On appeal,
the creator sends that `content_id`; the record's status flips to `under_review` and
a new audit row is appended — the original classification is never overwritten.

## Detection signals — what and why

Two signals were chosen because they capture **genuinely different properties** of
text: one *semantic*, one *structural*. Their errors are largely uncorrelated, so
the combination is more informative than either alone — and when they disagree, that
disagreement is itself useful (it drives results into the honest `uncertain` band).

### Signal 1 — LLM classifier (Groq), *semantic*
Asks `llama-3.3-70b-versatile` (temperature 0, JSON mode) to rate AI-likelihood in
`[0,1]` with a one-line rationale. It judges tone, hedging, register uniformity, and
idiosyncrasy holistically — the things a statistical metric can't see.
**Blind spot:** heavily edited AI, or AI told to imitate a specific voice, can fool
it; it is also non-deterministic and can be over-confident. Implemented in
[signals.py](signals.py) (`groq_signal`).

### Signal 2 — Stylometric heuristics (pure Python), *structural*
Three measurable properties, each mapped to an AI-likelihood contribution in `[0,1]`
then weighted-averaged (`stylo = 0.5·burstiness + 0.2·ttr + 0.3·punctuation`):
- **Burstiness** — coefficient of variation of sentence length. Humans vary sentence
  length a lot; AI is uniform. Low variation ⇒ higher AI-likelihood.
- **Type-token ratio** (100-word window) — lexical diversity.
- **Punctuation variety** — count of distinct "rich" punctuation types. Humans use a
  wider, messier range; AI leans on a narrow set.

**Blind spot:** short text (< 3 sentences) has too little signal, and deliberately
uniform human writing (technical docs, academic prose) scores AI-ish. Implemented in
[signals.py](signals.py) (`stylometric_signal`).

## Confidence scoring — what and why

The two signals are combined into a single `confidence ∈ [0,1]` (the system's
estimate that the text is AI-generated). See `combine_signals` in [app.py](app.py).

```
confidence = 0.6 · llm_score + 0.4 · stylo_score
```

The LLM gets the higher weight because it is the more informative signal on typical
prose; stylometry is the independent structural check that keeps it honest and lets
the system **degrade gracefully** — if the Groq call fails, stylometry carries the
call; if the text is too short for stylometry to be reliable (< 3 sentences), its
weight drops to 0.15 and the LLM carries it. A **short-text penalty** pulls thin
evidence (< 3 sentences or < 40 words) toward 0.5 so it lands in `uncertain` rather
than at a confident extreme.

**Attribution buckets** (deliberately *not* a binary flip at 0.5):

| confidence | attribution   |
|------------|---------------|
| ≥ 0.65     | `likely_ai`   |
| 0.35–0.65  | `uncertain`   |
| ≤ 0.35     | `likely_human`|

`certainty = 2·|confidence − 0.5|` is reported as a secondary transparency measure —
how far the result sits from a coin flip.

### Two example submissions with meaningfully different scores

Real outputs from Milestone-4 testing (see the calibration table below):

- **High-confidence AI** — *"Artificial intelligence represents a transformative
  paradigm shift…"* → **confidence 0.688** (llm 0.80, stylo 0.52) → `likely_ai`.
- **Lower-confidence / human** — *"ok so i finally tried that new ramen place
  downtown and honestly? underwhelming…"* → **confidence 0.249** (llm 0.20, stylo
  0.32) → `likely_human`.

A spread of **0.44** between clearly-AI and clearly-human text confirms the score is
meaningful variation, not a constant.

### Full calibration run (4 deliberately chosen inputs)

| Input | llm | stylo | confidence | attribution |
|-------|-----|-------|-----------|-------------|
| Clearly AI (paradigm-shift paragraph) | 0.80 | 0.52 | **0.69** | likely_ai |
| Clearly human (casual ramen review) | 0.20 | 0.32 | **0.25** | likely_human |
| Borderline: formal human (monetary policy) | 0.80 | 0.62 | **0.66** | likely_ai ⚠ |
| Borderline: lightly edited AI (remote work) | 0.40 | 0.41 | **0.42** | uncertain |

The formal-human case is a deliberate, documented false positive (see Limitations).

## Transparency labels

The label returned by `/submit` changes with the confidence score (see `build_label`
in [app.py](app.py)). `{pct}` = `round(confidence · 100)`. All three are reachable —
verified live:

- **High-confidence AI** (`likely_ai`):
  > ⚑ Likely AI-generated. Our analysis estimates a {pct}% chance this text was
  > produced by AI, based on two independent signals. If you believe this is wrong,
  > you can appeal using your content ID.

- **High-confidence human** (`likely_human`):
  > ✓ Likely human-written. Our analysis found no strong signs of AI generation
  > ({pct}% AI-likelihood). This is an estimate, not a guarantee.

- **Uncertain** (`uncertain`):
  > ? Inconclusive. Our signals disagree or are weak ({pct}% AI-likelihood, low
  > certainty). We are not confident either way; treat this result with caution and
  > appeal if it affects you.

## Appeals workflow

`POST /appeal` with `{ content_id, creator_reasoning }`:
1. Looks up the original classification by `content_id` (404 if unknown).
2. Appends an audit row with `status = under_review` and the `appeal_reasoning` —
   the original classification row is preserved, so the appeal is an *additional*
   timestamped fact, not an overwrite.
3. Returns a confirmation. `GET /appeals` is the reviewer queue of all appeals.

Automated re-classification is intentionally out of scope; a human reviewer resolves
the queue.

## Rate limiting

`Flask-Limiter` (in-memory storage) protects `/submit` at **`10 per minute; 100 per
day`** per IP. Reasoning: a real writer checking their own drafts submits a handful of
pieces in a sitting — 10/min is comfortably above that — while 100/day and the
per-minute cap stop a script from flooding the (paid, latency-bound) Groq endpoint.
`/appeal` is capped at 20/min.

**Evidence** — 12 rapid requests against the 10/min limit (clean window):

```
200 200 200 200 200 200 200 200 200 200 429 429
```

The first 10 succeed; requests 11–12 return **HTTP 429**.

## Known limitations

Named against a **specific property of the signals**, not generic "needs more data":

1. **Formal, uniform human writing is misclassified as AI.** Academic/technical prose
   has low sentence-length variance and narrow punctuation, which the *stylometric*
   signal reads as AI-like; the *LLM* signal also keys on "generic formal register."
   Both signals fail in the *same direction*, so the combination doesn't save it —
   the monetary-policy sample scored `confidence 0.66 → likely_ai` even though a human
   wrote it. Mitigation is partial: the wide `uncertain` band and short-text penalty
   catch some cases, but genuinely long formal human writing will still false-positive.
2. **Lightly edited AI text evades detection.** When a human rewrites AI output, the
   stylometric fingerprint becomes human-like and the LLM often stops smelling the AI
   register — both signals drift toward "human," so the system under-detects rather
   than landing in `uncertain`. This is inherent to signals that measure the *surface*
   of the text rather than its provenance.
3. **Short text is low-information.** Under ~3 sentences, stylometry is unreliable and
   the LLM has little to judge; the deliberate short-text penalty routes these to
   `uncertain` rather than risk a confident wrong call.

**If deploying for real:** calibrate thresholds on a labeled corpus (the 0.65/0.35
cutoffs are reasoned, not empirically tuned); add a third, provenance-based signal
(e.g. watermark/metadata detection) that doesn't rely on surface features; and log
reviewer overturn decisions to measure and correct the false-positive rate.

## Spec reflection

- **Where the spec helped:** writing the three label variants *verbatim* in
  planning.md before any UI work forced the uncertainty model to exist first. Because
  the "uncertain" label had to say something honest ("our signals disagree or are
  weak"), I designed a wide 0.35–0.65 band and a `certainty` measure up front, instead
  of discovering at build time that a 0.6 score had no sensible thing to display.
- **Where the implementation diverged:** planning.md originally defined the
  user-facing `confidence` as `certainty` (distance from 0.5). During Milestone 4 I
  changed `confidence` to be the **combined AI-likelihood** itself, keeping `certainty`
  as a secondary field. Reason: one number now both maps to the attribution buckets
  *and* matches the audit-log schema the assignment models (`confidence` alongside
  `llm_score`), which the two-definition approach made confusing. planning.md §3 was
  updated to record the change.

## AI usage

- **Instance 1 — scoring logic.** I gave the AI tool planning.md §2–§3 + the diagram
  and asked for `combine_signals`. Its first draft did a flat `0.5·llm + 0.5·stylo`
  average and bucketed with a hard 0.5 flip — which silently contradicted my spec's
  0.6/0.4 weights and 0.65/0.35 bands. I overrode it to match the spec and *added* the
  degrade-gracefully weighting and short-text penalty it had omitted.
- **Instance 2 — stylometric signal.** I asked it to generate `stylometric_signal`
  from the Signal-2 description. It produced sensible metric functions but combined
  them with equal weights and no reliability flag. I re-weighted toward burstiness
  (the most discriminating metric in my testing), added the `ok` flag for short text,
  and tuned the normalization thresholds after seeing that TTR barely moved on short
  samples.
- **Verification:** every signal was tested standalone on the four calibration inputs
  before wiring, and the generated scoring was checked against the planning.md
  thresholds (not pasted blindly).

## Setup

```bash
python -m venv .venv
source .venv/Scripts/activate      # Windows (Git Bash);  .venv/bin/activate on Mac/Linux
pip install -r requirements.txt
cp .env.example .env               # then add your GROQ_API_KEY
python app.py                      # serves on http://localhost:5000
```

> `pip install -r requirements.txt pip-system-certs --trusted-host pypi.org --trusted-host files.pythonhosted.org`.

# Submit
